Hardware One: An Overview

Hardware One is a custom embedded firmware platform built on the ESP32-S3 microcontroller using the ESP-IDF framework. It is designed to serve as a flexible, self-contained hardware hub that integrates sensing, communication, automation, and a web-based user interface into a single device. The project targets applications where a compact, capable device needs to operate in field or deployment conditions with minimal external infrastructure, while still offering rich remote management capabilities over a local network or direct wireless connections.

At its core, Hardware One runs on the ESP32-S3, a dual-core Xtensa LX7 processor with built-in WiFi and Bluetooth capabilities. The firmware takes full advantage of the chip's 8 MB of PSRAM and 16 MB of flash storage to support features that would be impossible on simpler microcontrollers. The system is organized into modular components: hardware abstraction layers for I2C peripherals, a virtual file system combining LittleFS for persistent storage with optional SD card support, a web server handling both a dashboard UI and a REST-style API, a command-line interface accessible over serial or via web terminal, and a growing set of communication subsystems including ESP-NOW for peer-to-peer mesh communication, BLE for proximity interactions, and MQTT for broker-based IoT messaging.

The device exposes a web dashboard accessible from any browser on the local network. This dashboard provides real-time sensor readings, GPS position, filesystem browsing and file transfer, device settings management, OTA firmware updates, and ESP-NOW peer pairing. Authentication is handled by a lightweight password-based login system. The web interface is served entirely from the device itself — no cloud dependency, no external CDN. Map tiles for GPS visualization are cached locally on the SD card, allowing offline use in the field.

Hardware One supports a broad range of I2C sensors and peripherals. The system dynamically discovers connected devices at startup via I2C bus scanning and instantiates drivers accordingly. Supported hardware includes an SSD1306 OLED display for local status output, a BME280 for environmental sensing, an ICM-42688-P IMU for motion and orientation, a VL53L1X time-of-flight distance sensor, a MLX90614 contactless thermometer, an APDS9960 gesture and proximity sensor, a GPS module providing NMEA sentence parsing and position fix tracking, a PCA9685 PWM driver for servo and LED control, and an Adafruit Seesaw gamepad for local input. NeoPixel LED strips can be connected and controlled with full RGB color and brightness management.

The CLI provides a persistent command history, an integrated help system, and the ability to register custom commands at runtime. Commands cover system diagnostics, sensor queries, network management, filesystem operations, LLM interaction, and automation control. The bonded device mode allows two Hardware One units to maintain a trusted relationship and route CLI commands between each other transparently, so a command entered on one device can execute on the other.

Automation on Hardware One is driven by a rule engine that evaluates conditions based on sensor values, time triggers, or external MQTT messages, and executes action chains in response. Conditions can be combined with logical operators. Actions include publishing MQTT messages, triggering GPIO outputs, updating display content, sending ESP-NOW messages to peers, or executing CLI commands. This system allows the device to respond autonomously to its environment without requiring a cloud service or an always-on host machine.

One of the more distinctive capabilities added to Hardware One is an on-device large language model inference engine. This engine loads a custom binary format called LLM1, which stores model weights in either full FP32 precision or INT8 quantized form with grouped scaling factors. The format supports both Llama-style architectures using RMSNorm and Rotary Positional Embedding, and GPT-2 style architectures using LayerNorm and learned positional embeddings with GELU activation. Model files are stored on the SD card and loaded into PSRAM at runtime. The inference engine implements the full transformer forward pass in plain C with no external ML library dependency. Sampling strategies include greedy, top-k, top-p, and mirostat v2 for adaptive perplexity-controlled generation. The web interface includes a dedicated LLM tab for prompt input and streamed output.

The LLM models that run on Hardware One are custom-trained small transformers. A companion Python toolchain handles training on TinyStories or domain-specific datasets, with several preset architectures tuned to fit within the device's PSRAM constraints. These presets range from a micro model at roughly 1 MB to stretch configurations pushing close to 8 MB. A browser-based converter tool takes a HuggingFace-format checkpoint and packages it into the LLM1 binary format with optional INT8 quantization, outputting diagnostic statistics to verify the conversion.

Hardware One is built with stability and field reliability as primary concerns. The firmware avoids dynamic heap allocation in hot paths, uses static buffers for string formatting, and applies defensive coding throughout network and filesystem operations. PSRAM is used for large allocations such as weight tables and audio buffers, keeping internal heap free for the RTOS and driver stack. The partition table allocates space for two application images to support OTA rollback. A debug file logging subsystem can write tagged log output to a file on the SD card for post-mortem analysis without requiring a serial connection.

The project uses ESP-IDF as its build system and RTOS foundation. FreeRTOS tasks are allocated per subsystem — the web server, sensor polling, automation engine, LLM inference, and communication layers each run in their own context with appropriate stack sizes and priority levels. The I2C bus has resilience logic to detect and recover from stuck-bus conditions. Settings are stored in a JSON file on LittleFS and include WiFi credentials, MQTT broker configuration, device identity, sensor calibration offsets, and feature enable flags. Settings can be exported and imported through the web interface for easy provisioning across multiple units.

Hardware One is intended to be a practical tool for hardware makers, field researchers, and IoT developers who want a capable standalone device without depending on cloud infrastructure. Its combination of local AI inference, mesh networking, rich sensing, and a complete web management interface makes it unusually self-sufficient for an embedded system of its size.
